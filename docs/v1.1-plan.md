- **F3** (cross-file overview) — **SHIP** with mitigations (§F3 below).

## Goals

1. **Preview-mode commenting** — let users add comments from the rendered Markdown preview, not just the source editor.
2. **Liveness** — reply/anchor updates written by an agent in another process should appear in VS Code without a manual focus-change.
3. **Cross-file overview** — a single panel summarizing all unresolved comments in the workspace.
4. **Production hardening** — address the remaining DA round-2 "should-fix" items that were deferred.

## Non-goals (still)

- Multi-user real-time collaboration (CRDT, presence, etc.).
- Server-side component. Everything stays local.
- Replacing the native VS Code Comments UI.

## Features

### F1 — Commenting in preview mode

**Why.** User asked for it. Reading a rendered preview is the natural review mode; forcing the reviewer to switch back to the source tab to leave a comment breaks flow.

**Approach.**
- Render Markdown ourselves into a webview (or post-process the built-in preview via `markdown.api.render`) and wrap every rendered block with `data-src-start` / `data-src-end` character-offset attributes sourced from the Markdown parser's `map` data (remark/markdown-it both expose this).
- In the webview, on text selection, read the nearest ancestor's `data-src-*` plus the offset inside the block's source range → compute `{start, end}` in the `.md` source.
- Post a message back to the extension via `Webview.postMessage`: `{ kind: "create-comment", uri, start, end, commentBody }`.
- Extension side opens the source doc (invisibly, if not already open) and calls the same `extractAnchor` + `sidecar.addComment` pipeline the editor path uses. No new storage format needed.
- Replies/resolve on existing threads: render a "💬 N" badge on each block that has comments; clicking opens a side panel with the thread (editable). Post edits back over the same channel.

**Complexity.** Medium-high. Source-map accuracy for nested constructs (tables, fenced code, HTML blocks) is the risky part.

**Ship behind a setting** `markdownCollab.enablePreviewComments` (default `false`) for a release, then flip after feedback.

**New files (sketch).**
- `src/previewCommentView.ts` — webview provider, source-map rendering, message router.
- `src/previewOffsets.ts` — pure helpers for translating DOM-range → source offsets; unit-testable.
- `media/preview.js`, `media/preview.css` — webview client.

### F2 — Live sidecar watcher

**Why.** Currently the plugin reloads on `onDidChangeActiveTextEditor` + on save. If the user is staring at the file while Claude Code writes a reply, they have to click elsewhere and back to see it. Round 1 DA explicitly cut this from v1 to remove race-condition complexity; now that the write-ordering and per-URI queue are solid, we can re-introduce it safely.

**Approach.**
- `vscode.workspace.createFileSystemWatcher("**/.markdown-collab/**/*.md.json")`.
- Ignore events whose path matches an entry in the self-write-token set (add one to `sidecar.saveSidecar` — a `Set<string>` of `<path>@<mtime>` tokens added just before rename, removed after the event fires).
- On external event: find the open doc whose sidecar this is; if any, `controller.reload(doc)` (already serialized per-URI from v1).
- Debounce per-path: 50ms. Prevents the temp-file-rename triggering two events.

**Complexity.** Low — the infra is already in place.

**DA-driven mitigations (required for ship):**
1. Self-write token keyed on **content hash** (SHA-256 of serialized JSON), not `<path>@<mtime>`. mtime is unreliable across APFS/NTFS/cloud-sync.
2. Debounce = **250ms** trailing edge (not 50ms). Covers Dropbox/iCloud write bursts and the tmp→rename two-event sequence.
3. Watcher handler must **skip reload if the per-URI queue is non-empty** (reuse `reloading` map in `commentController.ts`).
4. `handleReply` / `handleCreateThread` **must not dispose** the thread they're editing if a watcher reload arrives mid-edit. Patch anchors/replies in place instead.

### F3 — Cross-file overview panel

**Why.** Workspace-wide view of unresolved review state is missing. Currently the user can only see comments when the relevant file is focused.

**Approach.**
- New `vscode.TreeView` in the Explorer: "Markdown Review".
- Root items = files that have unresolved comments.
- Leaf items = each unresolved comment, label = anchor snippet, description = latest reply author + age.
- Click → open file, scroll to thread.
- Built from walking `.markdown-collab/**/*.md.json` with a cached scan that refreshes on the F2 watcher events.

**Complexity.** Low. Similar shape to the existing orphan TreeView.

### F4 — MCP server (optional)

**Why.** A stronger Claude Code integration. Skill-based reading works, but MCP gives Claude Code structured tools like `list_unresolved(file)`, `post_reply(commentId, body)`, and `update_anchor(commentId, ...)`, which are more reliable than prompting Claude to read/write JSON correctly.

**Approach.**
- Ship an MCP server binary with the extension: `node out/mcp/server.js`.
- Extension adds a command "Markdown Collab: Generate MCP config snippet" that outputs the JSON fragment for `~/.config/claude-code/mcp.json` (or equivalent).
- Tools exposed:
  - `list_comments(path: string, includeResolved?: boolean): Comment[]`
  - `add_reply(path: string, commentId: string, body: string): void` (author fixed to `"ai"`, `createdAt` auto-stamped).
  - `update_anchor(path: string, commentId: string, anchor: Anchor): void`.
- Reuses `src/sidecar.ts` directly — no duplicate storage logic.

**Complexity.** Medium. Shipping a subprocess with the extension has install friction; weigh vs. skill-only approach.

**Decision gate.** Ship F4 only if we hear Claude Code regularly mis-writes the sidecar JSON. Skill + file writes may be good enough.

### F5 — Anchor-matching v2

**Why.** v1 orphans any anchor that can't be found via exact + context + whitespace-normalized matching. Empirically, many near-misses are one-character edits (punctuation, trailing whitespace) that the user would want re-attached.

**Approach.**
- Add a bounded edit-distance matcher as a fourth step, with:
  - Minimum anchor length ≥20 non-whitespace chars.
  - Similarity threshold ≥0.92.
  - Only one candidate accepted; if two score above threshold within 0.02 of each other → still orphan.
- Show a "fuzzy-matched" badge on threads resolved this way so users know it's best-effort.

**Complexity.** Low-medium. Core library is small; the UX badge + test coverage is the bulk.

**Revised after v1 DA review rejected fuzzy matching as silent-mis-anchor risk** — mitigations here (badge, high threshold, tie-break rejection) address those concerns.

- Fuzzy match opt-in or opt-out? Leaning opt-in (`markdownCollab.enableFuzzyMatch` default `false`).
